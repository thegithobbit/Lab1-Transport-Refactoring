# Lab1-Transport-Refactoring: Рефакторинг + UML-моделювання

## Опис проєкту
Проєкт присвячений рефакторингу існуючої системи управління транспортним парком (однієї з попередніх лабораторних). Основна мета — перехід від процедурного підходу до чистої об'єктно-орієнтованої архітектури, яка відповідає принципам SOLID.

## Глосарій
* **Транспорт (Transport)** — базовий абстрактний клас, що визначає спільні властивості всіх засобів пересування.
* **Фабрика (TransportFactory)** — спеціалізований клас для створення об'єктів без прив'язки до їх конкретних класів у основному коді.
* **Поліморфна дія** — метод, який має спільну назву, але виконує різну логіку залежно від типу транспорту (наприклад, зліт для літака або дрифт для авто).

## Що саме було змінено (Рефакторинг)

Під час роботи було виділено кілька проблем у початковому коді ("legacy code"):
1.  **Жорстка залежність (Coupling):** Функція `main` містила забаго інформації про кожен тип транспорту.
2.  **Забагато перевірок типів:** Постійне використання `if-else` та `dynamic_cast` для виклику специфічних функцій.
3.  **Порушення принципу OCP:** Було не можливо додати новий транспорт без переписування `main.cpp` та логіки створення об'єктів.

**Виконані кроки:**
* **Впровадження патерну Factory:** Тепер об'єкти створюються централізовано через `TransportFactory.h`. Це дозволило очистити `main` від зайвої логіки.
* **Поліморфний інтерфейс:** Було додано віртуальні методи `performSpecialAction1()` та `performSpecialAction2()`.
* **Покращення інкапсуляції:** Додано геттери, що дозволило написати правильні Unit-тести для перевірки стану об'єктів.

## Аналіз результатів
Після рефакторингу код став значно чистішим (принцип **KISS**). Головна перевага — масштабність.

## Як запустити
1. Головна програма: `main.cpp`
2. Юніт-тести: `tests.cpp` (перевірка того, що рефакторинг не порушив базову логіку).
